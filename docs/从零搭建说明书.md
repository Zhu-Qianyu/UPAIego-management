# upaieasy! / UPAIego 管理端 — 从零搭建说明书

本文说明如何从**空环境**搭建本仓库所需的 **Supabase（PostgreSQL + Auth + Storage）**、执行全部 **SQL 迁移**，并启动 **Web 前端**（及可选 **CLI/后端**）。适用于新立项、新区域实例或灾难恢复后的重建。

**相关文档：** `DEPLOYMENT.md`（英文部署摘要）、`docs/DATABASE_SPECIFICATION.md`（表结构说明）、`docs/网页使用手册.md`（终端用户操作）。

---

## 1. 架构与组件

| 组件 | 作用 | 是否必需 |
|------|------|----------|
| **Supabase 项目** | PostgreSQL、Auth、`anon`/`service_role` Key、可选 Storage | **必需**（生产与标准联调） |
| **Web 前端** `frontend/` | React + Vite，浏览器直连 PostgREST | **必需**（管理界面） |
| **CLI / FastAPI** `backend/` | `DATABASE_URL` 直连 Postgres，设备注册与巡检 | 可选 |
| **板端 ROS2 Web Bridge** | 心跳写 `devices` 等 | 可选；需与 RLS/密钥策略一致 |

数据访问关系简述：

- 浏览器使用 **`anon` Key + 用户登录 JWT**，所有读写受 **RLS** 约束。  
- **切勿**把 `service_role` Key 写进前端环境变量。  
- 板端若需绕过用户 RLS，应使用服务端代理或单独策略，见 `docs/DATABASE_SPECIFICATION.md` 第 6 节讨论。

## 2. 准备工作

### 2.1 账号与工具

- [Supabase](https://supabase.com) 账号。  
- 本仓库已克隆到本地；可编辑文本与运行命令行。  
- **Node.js**（建议 LTS）用于前端；**Python 3** 用于后端/CLI（若使用）。  

### 2.2 建议保存的凭据（建项目后填写）

自 Supabase 控制台 **Settings → API** 复制：

- **Project URL**（形如 `https://<ref>.supabase.co`）  
- **anon public** key  

自 **Settings → Database → Connection string → URI** 复制连接串，并将 `[YOUR-PASSWORD]` 替换为数据库密码；CLI 建议使用 **Session pooler、端口 6543**（与 `DEPLOYMENT.md` 一致）。密码中含 `@` 等特殊字符时需 **URL 编码**。

## 3. 创建 Supabase 项目

1. 登录 Supabase → **New project**。  
2. 填写 **Project name**、**Database password**（务必安全保存）、**Region**（尽量靠近用户）。  
3. 等待实例就绪（约 1～2 分钟）。  

### 3.1 Auth（认证）

1. 打开 **Authentication → Providers**，按需启用 **Email** 等。  
2. **Authentication → URL configuration**：将 **Site URL** 设为将来用户访问前端的地址（本地开发可填 `http://localhost:5173`），避免邮件链接跳转错误。  

### 3.2 生产环境建议

- 在 **Authentication → Providers** 限制仅允许邮箱域名、或关闭自助注册，由管理员在控制台 **Users** 中邀请用户。  
- 定期备份数据库（Supabase 自带计划与导出能力以官方文档为准）。

## 4. 创建基础表 `public.devices`

在 Supabase **SQL Editor** 中**新建查询**，一次性执行以下脚本（在 `DEPLOYMENT.md` 基础上补全网站与板端常用列，与 `docs/DATABASE_SPECIFICATION.md` 推荐一致）。

```sql
CREATE TABLE IF NOT EXISTS public.devices (
    device_id            TEXT PRIMARY KEY,
    user_id              UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
    readable_name        TEXT NOT NULL UNIQUE,
    serial_id            TEXT,
    machine_id           TEXT,
    hostname             TEXT,
    registered_at        TIMESTAMPTZ,
    last_seen            TIMESTAMPTZ,
    calibration_status   TEXT NOT NULL DEFAULT 'pending',
    calibration_date     TIMESTAMPTZ,
    status               TEXT NOT NULL DEFAULT 'active',
    firmware_version     TEXT,
    notes                TEXT,
    calibration          JSONB
);

CREATE INDEX IF NOT EXISTS devices_user_id_idx ON public.devices (user_id);
```

> **说明：** 若曾仅执行过旧版 `DEPLOYMENT.md` 建表且缺少 `serial_id` / `calibration`，请用 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` 补列（见 `DATABASE_SPECIFICATION.md` 第 6 节）。

后续 **`docs/ROLE_SYSTEM_MIGRATION.sql`** 会为 `devices` 增加 `map_lat`、`map_lng`、`storage_used_mb`、`storage_capacity_mb`、`last_data_pickup_at` 等列，**无需**在本节手工添加。

## 5. SQL 迁移执行顺序（核心）

以下文件均在仓库 **`docs/`** 目录。请在 Supabase **SQL Editor** 中**每次打开完整文件、全选执行**（脚本内多含 `DROP POLICY`、函数替换，拆段执行易失败）。

### 5.1 总览表（按顺序）

| 顺序 | 文件 | 作用摘要 |
|------|------|----------|
| 1 | `ROLE_SYSTEM_MIGRATION.sql` | `profiles`、角色函数、注册触发器、`devices` 扩展列、管理台表、`scene_tasks` 等基础表、设备与 KPI 留言 RLS、`harvest_device` 初版 |
| 2 | `GROUP_TOPICS_BUSINESS_MIGRATION.sql` | 工作群、成员、话题、甲方需求、场景岗位、Storage 桶与策略、**第 9 节**替换 `devices` 的 fleet RLS 与 **带群范围校验的 `harvest_device`** |
| 3 | `GROUP_WORK_GROUPS_RLS_FIX.sql` | **可选但强烈推荐**：修复 `work_groups` / `group_members` 策略递归问题（若遇 infinite recursion 必跑） |
| 4 | `DEVICES_GROUP_SCOPE_RLS.sql` | **可选**：与第 2 步**第 9 节**内容高度重叠；若第 2 步已整文件执行成功，通常**无需再执行**；若你只有旧库需单独补 fleet 策略可执行 |
| 5 | `PROFILES_GROUP_PEER_SELECT.sql` | 同群成员可读他人 `profiles`（成员列表展示名等）；**依赖** `group_members` 已存在 |
| 6 | `SCENE_BUSINESS_ASSIGNMENT_MIGRATION.sql` | 场景业务 v2：`party_demands`/`scenario_positions` 扩展、`scene_tasks.group_id`、`scene_task_assignments`、`sync_scene_task_assignments` 等 |
| 7 | `PARTY_DEMAND_DEVICE_TYPE_MIGRATION.sql` | `party_demands.device_type` 列（可与 6 同批次，但须在 `party_demands` 存在之后） |
| 8 | `SCENE_TASKS_ADMIN_POSITION_BIND_MIGRATION.sql` | `scene_tasks.scenario_position_id`、同群校验、同步逻辑调整 |
| 9 | `SCENE_TASKS_BATCH_GENERATION_MIGRATION.sql` | RPC `batch_generate_scene_tasks_for_group` |
| 10 | `MANUAL_TRACKED_DEVICES_MIGRATION.sql` | 外部设备登记 `manual_tracked_devices` |
| 11 | `EXTERNAL_DEVICE_STATUS_MIGRATION.sql` | 外部设备状态由 `status_ok` 迁移为 `external_status`（**必须在**第 10 步之后） |
| 12 | `ADMIN_KPIS_ROLE_AND_PERIOD.sql` | `admin_kpis` 增加 `target_role`、`valid_from` / `valid_until` |
| 13 | `ADMIN_KPIS_METRIC_KEYS.sql` | 将 KPI `title` 规范为 `device_health_rate` / `scene_count` / `data_volume`（可与前端 `kpiMetrics.ts` 对齐） |
| 14 | `USER_ACCOUNT_DELETE_MIGRATION.sql` | `delete_own_account()`，供前端「注销账号」 |

### 5.2 执行注意

- **PostgreSQL 15+ / Supabase**：触发器语法为 `EXECUTE FUNCTION`；若报错可改为 `EXECUTE PROCEDURE`（见 `ROLE_SYSTEM_MIGRATION.sql` 内注释）。  
- 任一步报错后，应**阅读错误信息**；修复后多数脚本设计为 **可重复执行**（`IF NOT EXISTS`、`DROP POLICY IF EXISTS` 等），但请以各文件头部说明为准。  
- 执行 **`GROUP_TOPICS_BUSINESS_MIGRATION.sql`** 后，会自动处理 **Storage** 桶 **`scenario-workstation-snapshots`** 及 `storage.objects` 的 RLS 策略；无需在控制台手动建同名桶（除非脚本失败）。  

### 5.3 与「仅设备」旧库的衔接

若数据库里**已有** `DEPLOYMENT.md` 中的旧 RLS 策略名（如 `Users can read own devices`），`ROLE_SYSTEM_MIGRATION.sql` 会通过辅助函数批量删除 `devices` 上旧策略再创建新策略；按顺序执行即可。

## 6. 首个管理员与 `profiles`

### 6.1 注册触发器（推荐）

`ROLE_SYSTEM_MIGRATION.sql` 在 `auth.users` 上挂载 **`handle_new_user`**：新用户插入时，会根据 `raw_user_meta_data.role` 写入 `public.profiles`；无效或缺省则为 `device_operator`。

### 6.2 将首个用户提升为管理员

1. 在 **Authentication → Users** 邀请或自助注册一个账号并完成邮箱验证（若启用）。  
2. 在 **Table Editor → profiles** 找到该用户行，或将下列 SQL 中 UUID 换成该用户 `id`：

```sql
UPDATE public.profiles
SET role = 'admin', updated_at = now()
WHERE id = 'YOUR_USER_UUID_HERE';
```

3. 重新登录 Web，侧栏应出现 **管理台**、**全量设备** 等管理员菜单。

### 6.3 历史账号无 `profiles` 行

若用户早于触发器创建，可手工插入：

```sql
INSERT INTO public.profiles (id, role)
VALUES ('YOUR_USER_UUID_HERE', 'device_operator')
ON CONFLICT (id) DO UPDATE SET role = EXCLUDED.role, updated_at = now();
```

## 7. Web 前端

```bash
cd frontend
npm install
cp .env.example .env
```

编辑 **`.env`**：

- `VITE_SUPABASE_URL` = 项目 URL  
- `VITE_SUPABASE_ANON_KEY` = anon key  

可选：国内地图灰屏时配置 `VITE_MAP_TILE_URL` 等（见 `.env.example` 注释）。

```bash
npm run dev
```

浏览器访问终端提示的地址（默认 `http://localhost:5173`）。使用管理员账号登录后，依次验证：**管理台**、**群组管理**、**设备管理**、**场景业务**、**数采地图**（参见 `docs/网页使用手册.md`）。

生产构建：

```bash
npm run build
```

将 `dist/` 部署到 Vercel / Netlify / 自有 Nginx 等，并在托管平台配置相同的 `VITE_*` 环境变量。

```bash
cd backend
pip install -r requirements.txt
cp .env.example .env
```

在 **`.env`** 中设置 **`DATABASE_URL`**（建议使用 Supabase **pooler 6543** 连接串）。

本地测试可使用 SQLite，**与生产 Supabase schema 可能不一致**；生产问题以 Supabase 已执行迁移为准。

## 9. 搭建完成验收清单

| 序号 | 检查项 | 预期 |
|------|--------|------|
| 1 | 任意角色用户登录 | 无「未检测到账号角色」黄页（`profiles` 有行） |
| 2 | 管理员进入 **群组管理** | 可创建工作群、看到入群代码 |
| 3 | 第二账号用入群码申请 | 管理员可审批；通过后两账号同群可见话题 |
| 4 | 设备运维员 **注册设备** | `devices` 新增行，列表与详情可打开 |
| 5 | 场景业务员维护 **甲方业务**、**场景岗位** | 无 403；有 Storage 上传时文件进入对应桶路径 |
| 6 | 管理员 **一键批量生成草稿** / 发布任务 | 无 RPC 报错（需第 8～9 步迁移） |
| 7 | 外部设备 | 先甲方业务再登记；详情页登记编号 10 位十六进制（第 10～11 步） |
| 8 | 数采地图 **回收数据** | 仅满存储等允许时成功；失败检查 `harvest_device` 是否已部署 |
| 9 | **注销账号** | 弹窗确认后成功；失败检查 `USER_ACCOUNT_DELETE_MIGRATION.sql` |

## 10. 常见问题

| 现象 | 处理方向 |
|------|----------|
| `permission denied for table devices` | 未执行或未完整执行 RLS 相关迁移；从 `ROLE_SYSTEM` 起复查 |
| `infinite recursion detected in policy for relation work_groups` | 执行 `GROUP_WORK_GROUPS_RLS_FIX.sql` |
| Web 能注册设备但列表为空 | RLS 正常；检查当前用户 `user_id` 是否与插入行一致；管理员查「全量」与入群范围 |
| `relation "scene_task_assignments" does not exist` | 未执行 `SCENE_BUSINESS_ASSIGNMENT_MIGRATION.sql` 或顺序错误 |
| `batch_generate_scene_tasks_for_group` 不存在 | 执行 `SCENE_TASKS_BATCH_GENERATION_MIGRATION.sql`，且需第 8 步 |
| 外部设备仍见 `status_ok` / 无 `external_status` | 执行 `EXTERNAL_DEVICE_STATUS_MIGRATION.sql` |
| CLI `connection refused` / 认证失败 | 检查 `DATABASE_URL`、端口 **6543**、密码 URL 编码 |
| 地图空白 | 配置 `VITE_MAP_TILE_URL`（见 `网页使用手册` 与 `.env.example`） |
